# v1.4 corpora — external prompt-injection replication

This folder holds Cerberus v1.4.0's published numbers against three
**independent public prompt-injection corpora**, alongside the builtin
N = 5,100 baseline. Every report here is generated by the same harness
and the same code path that builds the published v1.4 baseline; the
only thing that changes is which corpus is loaded.

The point of this folder is to answer one question honestly:

> *Does the v1.4 baseline number generalise off the corpus the
> detectors were calibrated against?*

Short answer for v1.4.0: **block precision survives the move; block
recall does not.** Detail below.

## Headline table

All four runs use the same Cerberus 1.4.0 build, the same harness,
the same hardware, no operator-supplied weights, and the regex /
keyword / heuristic L2 path (the opt-in `cerberus-ai[ml]` extra is
not loaded — it ships without bundled weights by design).

| Corpus | N (atk / ben) | Block precision | Block recall | Block F1 | Benign FPR | mean μs | p95 μs |
|---|---|---|---|---|---|---|---|
| Cerberus builtin (v1.4 refresh) | 5,100 (1,700 / 3,400) | 69.1 % | **84.8 %** | 0.762 | 18.9 % | 1,240 | 1,500 |
| DeepSet `prompt-injections` | 662 (263 / 399) | **75.0 %** | 9.1 % | 0.163 | 2.0 % | 1,030 | 1,230 |
| Lakera `gandalf_ignore_instructions` (train) | 777 (777 / 0) | **100.0 %** | 30.1 % | 0.463 | n/a | 1,010 | 1,180 |
| Microsoft BIPIA — email task (composed) | 800 (600 / 200) | **75.0 %** | 10.0 % | 0.176 | 10.0 % | 1,540 | 1,820 |

`Block precision` = TP / (TP + FP) on the *block* decision.
`Block recall` = TP / (TP + FN) on the *block* decision.
`Benign FPR` is undefined for Gandalf because the train split has zero
benign cases by construction.

## What to read into this

1. **Latency holds across corpora.** p95 is 1.18–1.82 ms whether the
   input is a 30-character Gandalf jailbreak, a multi-paragraph BIPIA
   email, or a synthetic Cerberus case. The detection stack is
   genuinely deterministic-time at the millisecond scale.
2. **Precision survives the move off builtin** — actually goes up.
   75–100 % vs 69.1 % on builtin. When v1.4.0 says "block", it is
   right.
3. **Recall does not survive.** The regex / keyword / heuristic L2
   banks shipped in 1.4.0 catch 9–30 % of injections drawn from
   independent corpora vs 84.8 % on the builtin. **That gap is a
   corpus-overfit signal**, not a correctness claim. The honest read
   is: v1.4.0's L2 floor was calibrated against the builtin and does
   not generalise without an ML model loaded.
4. **The ML L2 path was not active for these runs.** v1.4 Delta #2
   (ML-backed prompt-injection classifier) is opt-in and ships
   without weights — operators point it at a model file they download
   separately. None of these runs had a model loaded; the numbers
   above are the regex-only floor.

The next obvious moves on the v1.5 roadmap are (a) seed L2 from these
external corpora rather than only from the builtin, and (b) ship a
public ONNX model under `cerberus-ai[ml]` so the ML-backed L2 path is
a real default rather than an opt-in stub.

## Reproducing

```bash
pip install cerberus-ai datasets

# DeepSet (Hugging Face) — 662 cases (263 attacks / 399 benign)
python -m cerberus_ai.validation.cli \
  --corpus deepset \
  --out out/deepset

# Lakera Gandalf ignore_instructions train split — 777 attacks
python -m cerberus_ai.validation.cli \
  --corpus gandalf \
  --out out/gandalf

# Microsoft BIPIA — email task, 600 attacks + 200 benign (seed=42)
python -m cerberus_ai.validation.cli \
  --corpus bipia \
  --out out/bipia
```

The named loaders fetch from source the first time and cache the
materialised JSONL under `~/.cache/cerberus/corpora/<name>/corpus.jsonl`
(override with `--corpus-cache <dir>` or
`$CERBERUS_CORPUS_CACHE`). Subsequent runs read the cache. To
re-fetch, pass `--force-refetch`.

`bipia` requires `git` to clone `microsoft/BIPIA`; `deepset` and
`gandalf` require the Hugging Face `datasets` package.

## Corpus shape (so the numbers can't be misread)

External corpora are pure prompt-injection benchmarks. They do *not*
exercise the L1 (privileged-data) or L3 (outbound-path) layers of
the trifecta; the loaders therefore label every case
`l1=l3=trifecta=False` and either `l2=True` (attack) or all-false
(benign). Reading the per-layer L1/L3 metrics on these corpora is
not meaningful — they are measuring "did Cerberus correctly say no
on a layer the corpus does not exercise", which is uninformative.
The relevant signal in this folder is **block** precision / recall,
which is what the table above leads with.

## Files

* `<corpus>/report.json` — canonical machine-readable output (drives
  CI gates and downstream tooling).
* `<corpus>/report.md` — release-note-ready markdown for each
  corpus.

The loader implementation lives at
`sdk/python/cerberus_ai/validation/external_loaders.py`. The CLI
wiring is in `cerberus_ai/validation/cli.py`. Tests are at
`sdk/python/tests/test_external_loaders.py`.
